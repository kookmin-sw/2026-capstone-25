## §1. 설계 원칙과 철학

### 1.1 핵심 명제

**맥락 기반 의미 분할**  
입력을 읽고 → 하나의 프로젝트로 이해하고 → 전체 맥락 상 나눠야 좋은 지점에서 부분 맥락을 잘 나눈다. 단계의 "크기"는 결과이지, 입력 조건이 아니다.

### 1.2 3대 설계 가치

| 가치 | 의미 |
|---|---|
| **의미 우선** | 도메인과 맥락이 먼저, 크기·시간은 그 다음. 파이프라인은 의미 경계를 따라 움직인다. |
| **사용자 주도** | 2차 쪼개기·재분해·직접 수정은 모두 사용자 선택. AI는 제안자, 결정자는 사용자. |
| **설명 가능성** | 모든 단계마다 왜 이 경계에서 끊었는지가 AI 추론 블록에서 드러난다. |

## §2. 웹 구조와 네비게이션

### 2.1 5탭 네비게이션

한발짝은 하단 네비게이션(모바일) 또는 좌측 사이드바(데스크탑)로 5개 탭을 전환한다.

| 탭 | 아이콘 | 핵심 기능 |
|---|---|---|
| **홈** | 🏠 | 할 일 추가 — 입력(직접/템플릿) → AI 분해 → 결과 확인 → 확정. 서비스의 핵심 플로우가 여기서 시작된다. |
| **일정** | 📅 | 캘린더 관리 — 확정된 단계를 날짜에 배정하고, 주간/월간 뷰로 일정을 조망한다. |
| **전체** | 📋 | 프로젝트 목록 — 모든 프로젝트를 한눈에 보고, 개별 프로젝트 상세(단계 체크·타이머·편집)로 진입한다. |
| **리포트** | ⏱ | 주간 분석 — 4주 추이 차트, 프로젝트별 시간 분배, 자동 코멘트(잘한 점/아쉬운 점)를 확인한다. |
| **나** | 👤 | 사용자 프로필 |

### 2.2 반응형 레이아웃

| 브레이크포인트 | 레이아웃 |
|---|---|
| ≤767px (모바일) | 헤더 상단 + 콘텐츠 스크롤 + 하단 네비 |
| 768–1023px (태블릿) | 동일 세로 배치 + 콘텐츠 폭 확장(720px) |
| ≥1024px (데스크탑) | 좌측 사이드바(248px) 네비 + 중앙 콘텐츠(960px) |
| ≥1440px (와이드) | 프로젝트 리스트 3열 그리드, 콘텐츠 1200px |
| ≥1800px (초대형) | 콘텐츠 1320px, 프로젝트 4열 그리드 |

### 2.3 헤더

- 좌측 — 발바닥 로고 + "한발짝" 텍스트.
- 우측 — "오늘 N분" 집중 시간 필(pill). 실시간 갱신.

## §3. 사용자 플로우 — 입력부터 4블록 결과까지

### 3.1 전체 흐름 개요

사용자는 한 번만 입력한다. 이후 서버가 5단 파이프라인을 지나 4블록 응답을 반환하고, 사용자는 결과 화면에서 각 단계를 펼쳐보며 필요에 따라 2차 쪼개기 · 재분해 · 직접 수정을 선택한다.

### 3.2 입력 화면

입력 화면은 **직접 입력**과 **템플릿** 두 가지 탭으로 나뉜다.

#### 직접 입력 탭

- **할 일 제목** (필수, 최대 28자 표시). 예: "졸업 논문 제출까지 전체 과정을 정리해 줘".
- **상세 메모** (선택, textarea). 추가 맥락, 제약 조건 등을 자유 입력.
- **첨부 파일** (선택, 최대 3개). 지원 형식:
  - 문서: **pdf · docx · txt · md**
  - 이미지: **png · jpg · webp · gif**
  - 서버에서 텍스트 추출 후 AI 입력에 결합.
- **시작일 / 마감일** (선택, date picker). D-Day 뱃지 계산에 사용.
- **AI 쪼개기 토글** (ON/OFF). OFF이면 단일 할 일로 저장, ON이면 AI 분해 파이프라인 진입.

#### 템플릿 탭

- 템플릿 카탈로그에서 원하는 워크플로우를 선택한다 (§8 참조).
- 선택 시 제목·설명·기간이 자동 채워지고, AI 분해에 `structureHint`가 전달된다.

#### 제출

- 단일 "할 일 추가하기" 버튼.
- 로딩 스피너는 INTAKE → DECOMPOSE → VALIDATE → PRESENT 단계를 미세하게 표시.
- 이전 입력값 자동 복원 기능 ("이전 입력값을 불러왔어요" 알림 + 닫기 버튼).

### 3.3 결과 화면 · 4블록 구조

**핵심 UI 원칙**: 단계 카드가 UI의 중심이다. 화면에 들어온 사용자는 "다음에 무엇을 하면 되는지"를 1초 안에 본다.

| 블록 | 내용 |
|---|---|
| **① 결과 블록** | 프로젝트명 + "✦ AI가 N개 단계로 정리했어요" 뱃지. 단계 리스트(기본 접힘). 각 단계를 펼치면 ‹가이드라인 · 하위 단계 · 쪼개기 버튼›이 나타난다. 단계 카드가 화면의 주된 시각 요소. |
| **② AI 추론 블록** | 접힘식 카드 "💭 왜 이렇게 나눴어요?". 펼치면 두 항목 표시: "무엇을 읽었는지"(분석 결과) + "어떤 기준으로 나눴는지"(경계 선택 근거). 통합 단일 호출의 "왜 이렇게 나눴는가"가 여기서 드러난다. |
| **③ 재분해 블록** | 3가지 방향 칩: "더 잘게"(10~20분 단위) · "더 크게"(1~2시간 블록) · "💬 AI에게 직접 얘기"(피드백 입력란 → 재생성). 개별 단계에 대한 "2단계 쪼개기" 버튼도 여기 포함. 최대 3개 이전 버전 히스토리 복원 가능. |
| **④ 확정·수정 블록** | "확정하기"(저장 및 전체 탭으로 이동) · "직접 수정하기"(인라인 편집 모드 진입) · "쪼개지 않고 저장"(단일 할 일로 전환) · "돌아가기"(입력 화면 복귀). |

### 3.4 전체 사용자 흐름

```
[홈 탭 · 입력] ─── 직접 입력 또는 템플릿 선택
       │
       ▼
[제출] ─── AI 쪼개기 ON → 로딩(INTAKE→DECOMPOSE→VALIDATE→PRESENT)
       │                    AI 쪼개기 OFF → 단일 할 일 저장 → 전체 탭
       ▼
[결과 화면 · 4블록] ─── 단계 확인 · 추론 펼침 · 가이드 펼침
       │
       ├─ 재분해 → 더 잘게 / 더 크게 / 피드백 → 결과 재생성
       ├─ 2단계 쪼개기 → 특정 단계 내부에서 3~5개 하위 단계 생성
       ├─ 직접 수정 → 드래그 순서변경 · 인라인 편집 · 추가/삭제
       │
       ▼
[확정] ─── 전체 탭 · 프로젝트 상세로 이동
       │
       ├─ 단계 체크(완료 표시)
       ├─ 타이머 실행(집중 시간 기록)
       └─ 일정 배정(캘린더에 날짜 할당)
```

## §4. AI 분해 논리 — 맥락 기반 의미 분할

### 4.1 두 레이어 · 하나의 호출

- **레이어 A** — "어디서 끊는가" · 6 신호(Division Signals).
- **레이어 B** — "각 단계를 어떻게 기술하는가" · 3 규칙(Quality Rules).

두 레이어는 독립적으로 작동하며, 둘 다 충족되어야 좋은 분해다. 경계가 잘 잡혔어도 각 조각의 표현이 흐리면 실행에 이르지 못한다.

**핵심**: 두 레이어는 하나의 AI 호출 안에서 함께 작동한다. DECOMPOSE 단계에서 **Claude Sonnet 4**가 입력을 한 번 읽으면서 (a) 프로젝트 분석(유형·목표·현재 phase·제약)과 (b) 6 신호 기반 경계 탐색을 동시에 수행하고, 그 경계로 나뉜 각 단계를 3 규칙에 맞게 기술한다. "먼저 이해한 뒤 나눈다"가 아니라 "이해하면서 나눈다"이다.

### 4.2 레이어 A · 6 신호 (경계 탐색)

| 신호 | 의미 | 예시 |
|---|---|---|
| ① **Phase 전환** (Phase Boundary) | 프로젝트 진행 단계(phase)가 바뀌는 지점. 맥락상 가장 강한 경계로 작동한다. | 논문에서 "선행 연구 조사 → 방법론 설계" |
| ② **산출물 단위** (Artifact Boundary) | 관찰 가능한 결과물이 손에 쥐어지는 지점. "뭔가 끝냈다"는 감각을 만드는 자연 경계. | "개요 한 장", "초록 초안", "핵심 도표" |
| ③ **판단·결정 지점** (Decision Point) | 다음 단계의 방향을 정하는 선택이 필요한 지점. 결정 앞에서 끊어 숙고 여지를 남긴다. | "세 방법론 중 하나 결정" 앞에서 분리 |
| ④ **사고 모드 전환** (Cognitive Mode Shift) | 인지 모드가 바뀌는 지점 — 읽기↔쓰기, 분석↔발산, 이해↔판단 사이에서 끊는다. | "선행 연구 읽기" → "선행 연구 정리하기" |
| ⑤ **의존성 완결** (Dependency Closure) | 앞 단계가 끝나야 다음 단계가 가능한 최소 경계. 병렬 가능은 묶지 않고 "순서 무관" 태그로 표시. | "데이터 수집 완료" → "분석 시작" |
| ⑥ **실행 환경 전환** (Context Switch) | 도구·장소·시간대가 바뀌는 지점. 이동·준비 비용이 발생하는 지점에서 묶어 끊는다. | "도서관 자료 수집" → "노트북 집필" |

### 4.3 레이어 B · 3 규칙 (단계 기술)

| 규칙 | 의미 | 리라이트 예 |
|---|---|---|
| **구체성** · Specificity | 동사 + 구체 대상 + 가시 산출물을 모두 가진다. 모호 동사("연구하기", "공부하기" 단독)는 prompt 단계에서 자동 리라이트. | "연구하기" → "핵심 논문 3편 초록·결론 정리" |
| **즉시성** · Immediacy | 지금 · 혼자 · 바로 시작 가능. 외부 대기(피드백·승인 등)는 별도 단계로 만들지 않고, 대기를 유발하는 사용자 액션 단계의 description에 후속 행동을 흡수한다. | "초안을 보낸 뒤 3일 안에 답이 없으면 재발송" 한 줄을 description에 포함 |
| **판정 가능성** · Verifiability | 완료 상태가 파일·메시지·제출·체크·수치 중 하나로 관찰 가능. 주관 부사("충분히", "잘")는 prompt 단계에서 자동 리라이트. | "충분히 공부" → "3장 연습문제 10개 풀기" |

### 4.4 통합 단일 호출 설계

이전 설계는 "UNDERSTAND(이해) → PREPARE(도메인 문법 주입) → DECOMPOSE(분해)"의 3단 분업이었다. 통합 설계에서는 이 셋을 하나의 DECOMPOSE 호출로 합친다. 이유는 세 가지다.

- **도메인이 미리 만든 "문법"에 갇히지 않는다** — 실제 입력은 2~3개 도메인이 섞인 경우가 많은데, 이를 15~20개 고정 카테고리 중 하나로 강제 분류하면 오히려 분해 품질을 떨어뜨린다.
- **프롬프트가 단순해진다** — 고정 접두부(시스템 + 6 신호 + 3 규칙 + JSON 스키마)가 모든 호출에서 동일하므로 접두부 파편화 없이 Anthropic Prompt Cache 적중률이 극대화된다 (입력 토큰 90% 할인).
- **Claude Sonnet 4가 충분히 똑똑하다** — 짧은 분류 호출을 따로 둘 필요 없이, 한 번의 호출에서 분석(analysis)과 분해(steps)를 구조화 JSON으로 함께 산출할 수 있다.

AI의 출력은 아래와 같은 통합 JSON이다. analysis 섹션이 먼저 오고, 그에 근거해 steps 배열이 이어진다.

- `analysis`: `{ primary_type, secondary_tags, goal, current_position{ phase_label, phase_index }, constraints, needs_clarification, confidence }`
- `steps`: `[ { id, parent_step_id, title, description, guide, first_move, unblocker, estimated_minutes, boundary_signal, done, time_spent } ]`

## §5. 5단 파이프라인과 VALIDATE

### 5.1 파이프라인 개요

입력은 1회, 서버 내부는 5단. DECOMPOSE 하나의 AI 호출이 분석(유형·목표·현재 phase·제약)과 분해(6 신호 경계 + 3 규칙 기술)를 함께 수행한다. VALIDATE는 AI가 코드보다 못하는 일(식별자 일관성, 분해 성립)만 책임지는 가벼운 검증층이다.

| 단계 | 주체 | 하는 일 | 산출/비고 |
|---|---|---|---|
| ① **INTAKE** | 서버 | 텍스트·파일 수용, pdf·docx·txt·md·이미지 추출, 빈·과다 입력 차단, 템플릿 바인딩 처리 | `{ rawText, meta, templateHint }` · AI 호출 없음 |
| ② **DECOMPOSE** | AI (단일 호출) | 분석(유형·목표·현재 phase·제약) + 분해(6 신호 경계 + 3 규칙 기술)를 통합 JSON으로 산출 | Claude Haiku 4.5 · 평균 3~6초 · 고정 접두부 캐시 |
| ③ **VALIDATE** | 서버 | 분해 성립(최소 2단계) + step id 유일성을 결정적으로 검증 | blocker 발견 시 직전 issue를 user 메시지에 실어 1회 자동 재호출 |
| ④ **PRESENT** | 서버 | 4블록 응답 포장, `validation` 블록 동봉, 가이드라인 생성, DB 기록 | `{ 결과, 추론, 검증, 재분해, 확정 }` |
| ⑤ **REFINE** | 사용자 주도 | 2차 쪼개기·재분해·직접 수정 요청 시 DECOMPOSE로 재진입 | 접두부 캐시 재사용 → 입력 토큰 90% 할인 |

### 5.2 VALIDATE 책임 분담

검증은 "AI가 코드보다 못하는 일"만 코드가, "코드가 AI보다 못하는 일"은 prompt가 책임진다. 의미 품질(모호 동사·산출물·주관 부사) 판정은 한국어·영어 혼용·문맥 의존이 강해 결정적 검사로 풀기 어렵고 위양성을 만들기 쉬워, 시스템 프롬프트의 6 신호 · 3 규칙 가드에 위임한다.

**코드 책임 — 결정적 무결성 검사 (`validate/index.ts`)**

- **분해 성립** (`no_decomposition`) — 단계가 2개 미만이면 "분해가 일어나지 않았다"고 판정. 분해 서비스의 핵심 계약을 지키는 게이트.
- **step id 유일성** (`duplicate_id`) — 같은 응답 안에 동일 `id`가 둘 이상 부여되지 않았는가. React key·DB 외래키·`parent_step_id` 매핑 등 후속 무결성을 지킨다.

두 검사 모두 `severity: "blocker"`이며, 하나라도 발견되면 직전 issue 메시지를 다음 user 메시지에 동봉해 DECOMPOSE를 1회 재호출한다 (의미 재시도). 그 외 발견은 응답의 `validation.issues`에 노출만 되고 자동 재시도를 트리거하지 않는다 (warning 정책).

**Prompt 책임 — 의미 품질 가드**

- **6 신호** — 어디서 끊을지(phase·artifact·decision·mode·dependency·context).
- **3 규칙** — 각 단계가 구체성·즉시성·판정 가능성을 갖췄는지. 위반이 발견되면 모델이 자동 리라이트.
- **id 부여 규칙** — `s1, s2, s3`… 형태로 1부터 순서대로, 같은 응답 안에서 절대 중복 없이.
- **외부 대기 흡수** — "기다림" 단계를 따로 만들지 않고, 대기를 유발하는 사용자 액션 단계의 description에 후속 행동을 한 줄로 적어 흡수한다.
- **활동 중복 금지** — 같은 활동을 다른 표현으로 두 번 만들지 않는다.

### 5.3 자동 재시도 정책

DECOMPOSE 호출은 같은 요청에서 최대 3회까지 일어난다.

1. **1회차** — 정상 호출.
2. **파싱 실패 시 2회차** — `extractJsonObject` · `JSON.parse` · zod schema 중 어디서든 실패하면 `null`을 반환. 같은 입력으로 1회 재시도.
3. **검증 blocker 시 3회차** — `validate()`가 blocker를 반환하면 issue 메시지를 user 메시지에 동봉해 1회 재시도. 재시도 결과도 다시 validate를 거쳐 issues가 갱신된다.

prompt cache 적중 덕분에 2~3회차의 입력 토큰은 1회차 대비 약 10% 비용. 재시도가 또 실패하면 마지막 정상 결과를 그대로 응답하고, `validation.ok = false`로 클라이언트에 신호한다 — 무한 루프 방지.

## §6. 석사 논문 예시 — 1차 분해와 2차 분해

사용자 입력 가정: "AI를 활용한 할 일 분해 서비스에 대한 석사 논문을 써야 해. 지금 주제 정도만 있고, 3개월 뒤 중간 발표, 6개월 뒤 최종 제출이야."

### 6.1 1차 분해 (이해 단위 10개)

AI는 이 입력을 한 번의 DECOMPOSE 호출로 처리한다. analysis 섹션에서 `primary_type="학술·논문"`, `goal="할 일 분해 서비스에 대한 석사 논문 완성"`, `current_position.phase_label="주제 착안"` 등을 산출하고, 그에 근거해 6 신호로 경계를 찾는다. 각 단계는 이해 단위(Mental Chunk, 1~3시간 체감)지만 체크리스트 기반 완료가 가능하도록 기술된다.

| # | 단계 | 완료 기준 (판정 가능) | 적용 신호 |
|---|---|---|---|
| 1 | 연구 주제 구체화 | 연구 질문 1문장 확정 + 핵심 키워드 5개 목록 파일 | 산출물 단위 |
| 2 | 선행 연구 범위 설정 | 검색어 조합 3가지 · 검색할 DB 3곳 선정 메모 | 판단·결정 지점 |
| 3 | 선행 연구 읽기 | 핵심 논문 5~8편의 초록·결론·방법론 노트 | 사고 모드(읽기) |
| 4 | 선행 연구 정리 | 비교 표 + 본인 연구의 차별점 2문단 초안 | 사고 모드 전환(읽기→쓰기) |
| 5 | 연구 방법론 결정 | 3후보 비교 후 1개 확정 + 선택 근거 1문단 | 판단·결정 지점 |
| 6 | 데이터/실험 설계 | 프로토타입 명세 · 평가 지표 3개 정의 | 산출물 단위 |
| 7 | 중간 발표 자료 | 10~12장 슬라이드 + 발표 스크립트 초안 | 산출물 · 실행 환경(발표) |
| 8 | 본 집필 (이론/방법) | 1~3장 초안 문서 | 산출물 단위 |
| 9 | 본 집필 (결과/고찰) | 4~5장 초안 + 그림·표 삽입 | 산출물 단위 · 의존성 완결 |
| 10 | 최종 검토·제출 | 교정 체크리스트 완료 + 지도교수 피드백 반영 + 제출 | 산출물 · 의존성 완결 |

### 6.2 2차 분해 예시 — ③ "선행 연구 읽기" 펼치기

사용자가 이해 단위 ③을 선택하고 "2단계 쪼개기" 버튼을 누른다. AI는 동일 6 신호·3 규칙으로 이 단계 내부에서만 다시 경계를 잡고, 실행 단위(50분 이내 지향)로 기술한다.

| # | 실행 단위 | 완료 기준 |
|---|---|---|
| 3-a | 검색어로 논문 20편 추리기 | zotero 라이브러리 20건 저장 · 각 논문 태그 3개 부착 |
| 3-b | 초록 읽고 상위 8편 선정 | 선정 이유 메모가 붙은 8건 하이라이트 리스트 |
| 3-c | 핵심 논문 3편 정독 노트 | 논문별 1쪽 요약 노트 3개 (문제·방법·결과·한계) |
| 3-d | 보조 논문 5편 스킵 리딩 | 5편 × 3줄 요약 표 |
| 3-e | 인용 관계 지도 작성 | 노드 8개·엣지 주요 관계 그래프 파일(.svg 또는 손글씨 사진) |

**2차 분해의 의미**: AI가 "이해 단위"까지만 책임진다. 사용자가 실행 감각을 원할 때만 "실행 단위"로 더 내려간다. 이 선택권은 항상 사용자에게 있다.

## §7. 결과 화면 · 가이드라인 상세

### 7.1 단계 카드 구조

- **접힌 상태** — 단계 번호(원형 뱃지), 제목, 예상 시간 뱃지.
- **펼친 상태** — 제목 아래에 가이드라인 섹션 + 하위 단계(있을 경우) + "2단계 쪼개기" 버튼.
- **가이드라인 섹션** — 단계를 펼칠 때 lazy-load로 생성. 실행 방법을 구체적으로 안내하는 1~2문장.

### 7.2 가이드라인의 3문장 템플릿

가이드라인은 가능한 한 아래 3문장 구조를 따르되, UI에서는 하나의 `guide` 텍스트로 통합 표시한다.

| 문장 | 작성 규칙 |
|---|---|
| **산출물** (Output) | 완료 후 무엇이 있어야 하는지를 "파일·메시지·수치" 형태로 한 줄. |
| **첫 동작** (First Move) | 지금 1분 안에 시작할 수 있는 가장 작은 행동 한 줄. 미루기 방지 장치. |
| **막힘 대응** (Unblocker) | 가장 흔한 막힘 한 가지와 그때의 대안 한 줄. 의사결정 피로 감소. |

### 7.3 가이드라인 예시 — "핵심 논문 3편 정독 노트"

| | |
|---|---|
| **산출물** | 논문별 1쪽 요약 노트 3개 (파일명: paper1.md, paper2.md, paper3.md). 각 노트에 문제·방법·결과·한계 4문단. |
| **첫 동작** | 논문 3편 파일을 책상 위에 펼치고, paper1.md 파일을 새로 만들어 제목만 먼저 적는다. |
| **막힘 대응** | 영어 논문이 막히면 결론 문단부터 역순으로 읽는다. 이해 안 되는 수식은 "나중에 확인" 주석으로 표시하고 지나간다. |

## §8. 템플릿 시스템

### 8.1 개요

사용자가 "빈 종이에서 시작"하는 부담을 줄이기 위해, 자주 쓰이는 할 일 유형을 사전 정의 템플릿으로 제공한다. 템플릿은 입력을 안내할 뿐, AI 분해를 대체하지 않는다 — `structureHint`로 AI에 맥락을 전달하되, 최종 분해는 여전히 6 신호·3 규칙으로 수행된다.

### 8.2 카테고리 · 8종

| 카테고리 | 아이콘 | 예시 템플릿 |
|---|---|---|
| **학업** | 📚 | 기말 레포트 작성, 시험 준비, 발표 자료 제작, 조별 과제 |
| **개발** | 💻 | 사이드 프로젝트, 포트폴리오 사이트, 버그 수정 스프린트, API 서버 구축 |
| **글쓰기** | ✍️ | 블로그 포스팅, 자기소개서 작성, 뉴스레터 기고 |
| **취업** | 💼 | 이력서 정리, 면접 준비, 포트폴리오 정리 |
| **창작** | 🎨 | 유튜브 영상 제작, 일러스트 작업, 음악 커버 |
| **생활** | 🏠 | 이사 준비, 대청소, 요리 레시피 정리 |
| **여행** | ✈️ | 여행 계획 세우기, 짐 싸기 체크리스트 |
| **건강** | 💪 | 운동 루틴 만들기, 식단 관리 시작 |

### 8.3 템플릿 데이터 구조

```typescript
interface Template {
  id: string;                    // 예: "study-report"
  category: string;              // 예: "학업"
  name: string;                  // 예: "기말 레포트 작성"
  icon: string;                  // 이모지
  summary: string;               // 한 줄 설명
  prefill: {
    title: string;               // 제목 자동 채움
    suggestedDays: number;       // 권장 소요일
    noDueDate?: boolean;         // 마감일 없는 유형
  };
  customFields: string[];        // 사용자 입력 플레이스홀더 (메모에 채움)
  scaleHint: "small" | "medium" | "large";
  structureHint: string;         // AI에 전달할 분해 가이드
  previewChunks: string[];       // 미리보기용 5개 예시 단계
  featured?: boolean;            // 추천 템플릿 여부
}
```

### 8.4 템플릿 미리보기

템플릿 카드를 탭하면 바텀시트가 올라오며 다음을 표시한다:

- 아이콘 + 이름 + 한 줄 요약
- 메타 정보 행: 규모(소/중/대) · 권장 소요일
- 예시 단계 5개 (번호 + 제목)
- "이 템플릿으로 시작" 버튼

### 8.5 템플릿 적용 흐름

1. 사용자가 템플릿을 선택하면 입력 화면 상단에 템플릿 뱃지 표시.
2. 제목·메모 칸에 `prefill` 값이 자동 채워진다.
3. `customFields`가 있으면 메모 칸에 플레이스홀더로 안내.
4. 시작일은 오늘, 마감일은 `suggestedDays` 후로 자동 설정.
5. AI 호출 시 `structureHint`가 시스템 프롬프트에 추가되어 분해 품질을 높인다.
6. 결과 화면에서 템플릿 출처 뱃지("📋 템플릿 기반") 표시.

## §9. 타이머 · 포모도로

### 9.1 개요

각 단계(chunk)에서 "집중 시작" 버튼을 누르면 타이머 화면으로 진입한다. 시간을 측정하며 집중하고, 완료 시 해당 단계에 소요 시간이 누적된다.

### 9.2 타이머 흐름

```
[단계 카드] → [시간 설정] → [카운트다운] → [완료]
                                 ↕
                              [일시정지]
```

### 9.3 시간 설정

- **기본값**: 25분 (포모도로 1세트).
- **빠른 프리셋 버튼**: 10분 · 20분 · 30분.
- **사용자 지정**: 분·초 스피너로 1~180분 범위 설정.
- 시·분 단위로 ▲/▼ 버튼을 눌러 조절.

### 9.4 카운트다운 화면

- **원형 프로그레스 링** (240×240px): 남은 시간을 시각적으로 표시.
- **중앙 숫자**: 분:초 표시 (탭 제목에도 동기화).
- **상태 텍스트**: "FOCUS" (집중 중) / "PAUSED" (일시정지).
- **하단 버튼 2개**: 일시정지/재개 · 종료.

### 9.5 완료 처리

- 타이머 종료(자연 만료 또는 수동 종료) 시 완료 화면 표시.
- 집중 시간이 해당 단계의 `timeSpent`에 누적 저장.
- 헤더의 "오늘 N분" 필에 즉시 반영.
- "한 단계 마무리!" 축하 메시지 + 돌아가기 버튼.

### 9.6 포모도로 사이클

| 사이클 | 시간 | 설명 |
|---|---|---|
| 집중 | 25분 | 카운트다운 + FOCUS 표시 |
| 짧은 휴식 | 5분 | 4회 반복 후 긴 휴식 |
| 긴 휴식 | 15분 | 4세트(= 2시간) 완료 후 |

## §10. 프로젝트 관리 — 전체 탭 · 프로젝트 상세

### 10.1 개요

전체 탭은 사용자가 매일 들르는 공간이다. 확정된 모든 프로젝트가 한 화면에 모이고, 카드를 누르면 해당 프로젝트의 단계별 진행 상황을 들여다보는 상세 화면으로 들어간다. 두 모드는 같은 탭 안에서 전환된다.

| 모드 | 무엇을 보여주는가 | 주된 행동 |
|---|---|---|
| **목록 모드** (기본) | 모든 프로젝트를 마감일 기준으로 묶어 카드 리스트로 표시 | 다음 단계 시작, 카드 클릭 → 상세, 삭제 |
| **상세 모드** | 한 프로젝트의 모든 단계와 진행률 | 단계 펼치기 · 시작 · 완료 체크 · 2단계 쪼개기 · 수정 · 삭제 |

핵심 명제는 **"다음에 뭘 하면 되는지"가 카드와 단계 양쪽 첫 줄에 보인다**이다. 사용자는 들어와서 1초 안에 다음 행동을 발견해야 한다.

### 10.2 전체 탭 — 목록 모드

#### 10.2.1 헤더와 메트릭

화면 상단에 두 요소가 고정된다.

- **타이틀** — "나의 할 일".
- **메트릭 두 개** — `진행중 N` · `완료 N`. 진행중은 진행률 100% 미만, 완료는 100%인 프로젝트 수.

완료된 프로젝트도 목록에서 숨기지 않는다. 사용자가 직접 삭제하기 전까지 남는다(과거 성취감 보존 · 회고 가능).

#### 10.2.2 D-Day 그룹핑

카드는 마감일별로 그룹화된다.

- **마감일 있는 프로젝트** — 마감일 오름차순(가까운 순)으로 정렬, 동일 날짜끼리 한 그룹.
- **마감일 없는 프로젝트** — "마감 X" 그룹으로 별도 묶임, 항상 맨 아래.
- **그룹 헤더** — D-Day 뱃지(빨강·주황·검정으로 긴급도 표시) + 날짜 텍스트.

이 그룹핑은 ADHD 사용자가 "지금 가장 급한 게 뭐지?"를 시각적으로 즉시 파악하게 만드는 장치다.

#### 10.2.3 프로젝트 카드 — 일반 (분해된 프로젝트)

| 영역 | 내용 |
|---|---|
| **상단** | 프로젝트 색상 점 + 프로젝트명 + 우측 화살표(상세 진입 표시). 완료 시 라인 스루 + "완료" 뱃지. |
| **중단** | 진행률 바(프로젝트 색) + 우측 % 텍스트(굵게). |
| **하단 — 다음 단계 박스** | 회색 박스 안에 "다음 할 일" 텍스트 + "시작 →" 버튼. 버튼 클릭 시 시간 설정 화면으로 직행. |
| **우상단** | 호버 시 빨강으로 변하는 삭제 버튼. |

#### 10.2.4 프로젝트 카드 — 단일 (분해 안 한 프로젝트)

AI 쪼개기를 OFF로 저장한 프로젝트(또는 결과 화면에서 "쪼개지 않고 저장")는 다른 카드 형태를 가진다.

- **체크박스 한 개**가 카드 좌측에 위치 — 체크하면 즉시 완료.
- 진행률 바·다음 단계 박스 없음.
- 카드 클릭으로 상세 진입은 안 함(단일이라 상세할 게 없음).
- 우측에 "시작 →" 버튼(타이머로 진입).

#### 10.2.5 빈 상태

확정된 프로젝트가 하나도 없을 때:

- 📋 이모지 + "아직 할 일이 없어요" 타이틀
- "홈에서 첫 번째 할 일을 만들어보세요" 안내 문구

홈 탭으로 유도하는 미세한 힌트.

#### 10.2.6 카드 인터랙션

| 행동 | 결과 |
|---|---|
| 카드 본체 클릭 | 상세 모드 진입 (단일 카드는 제외) |
| "시작 →" 버튼 | 다음 단계의 시간 설정 화면으로 직행 (목록을 떠나지 않고 빠른 실행) |
| 삭제 버튼 (휴지통 아이콘) | 확인 후 프로젝트 영구 삭제 |
| 단일 카드 체크박스 | 즉시 완료/완료해제 토글 |

### 10.3 프로젝트 상세 모드

카드 클릭 시 같은 탭 안에서 상세 화면으로 전환된다(라우트는 변경되지만 사이드 패널은 유지).

#### 10.3.1 헤더와 진행률 카드

- **헤더** — 좌측 ← 뒤로가기 버튼 + 프로젝트명(큰 글자) + 그 아래 기간 텍스트(`시작일 ~ 마감일`) + D-Day 뱃지.
- **진행률 카드** — 진행률 % 바 + 우측에 큰 숫자로 `완료수 / 전체수`.

진행률은 단계의 `done` 비율로 자동 계산되며, 하위 단계가 있는 단계는 모든 하위가 완료돼야 본 단계가 완료로 친다.

#### 10.3.2 단계 카드 (접힌 상태)

진행률 카드 아래에 "단계별 진행 상황" 라벨, 그 아래로 단계 카드들이 세로로 나열된다.

- **번호 뱃지** — 원형, 단계 순서. 다음 단계는 강조 그라데이션 배경 + 흰 숫자.
- **제목** — 완료된 단계는 라인 스루 + 회색.
- **상태 텍스트** — "예상 25분" / "30분 진행" / "완료" 등.
- **펼침 버튼** — 우측 ▼ 아이콘. 다음 단계는 자동으로 펼침 상태.

#### 10.3.3 단계 카드 (펼친 상태)

펼치면 카드 하단에 다음이 나타난다.

| 영역 | 내용 |
|---|---|
| **가이드라인** | 회색 배경 박스. §7.2의 3문장(산출물·첫 동작·막힘 대응)을 한 단락으로 표시. |
| **하위 단계 리스트** (있을 때만) | 2차 분해 결과를 박스 안에 누적. 각 항목은 번호 + 제목 + 상태 + "시작" 버튼 + 완료 체크. |
| **액션 버튼 행** | 하위가 없으면 `2단계 쪼개기` + `시작` + 완료 체크. 하위가 있으면 `세부 단계 수정` + 완료 체크. |

#### 10.3.4 단계별 액션 — 세 갈래

| 액션 | 호출 시점 | 효과 |
|---|---|---|
| **시작** | 다음 단계, 또는 펼쳐진 단계 | 시간 설정 화면 → 타이머(§9)로 전환 |
| **2단계 쪼개기** | 하위 단계가 없는 카드 | DECOMPOSE 재호출 → 해당 단계 내부에서만 3~5개 하위 단계 생성 (§6.2 참조) |
| **완료 체크** | 어느 단계든 | `steps.done` UPDATE → 진행률 즉시 갱신, 다음 단계 자동 강조 이동 |

#### 10.3.5 하위 단계(2차 분해) 표시

특정 단계에 하위가 생성되면 펼친 상태에 박스로 누적된다.

- 각 하위: 작은 번호 + 제목 + 상태 + 우측에 `시작` 버튼과 완료 체크.
- 하위가 모두 완료되면 본 단계가 자동 완료로 마킹.
- 하위 단계는 "세부 단계 수정"에서 추가·삭제·순서 변경 가능.

이 구조 덕에 사용자는 "이해 단위(1차)"와 "실행 단위(2차)" 사이를 자유롭게 오갈 수 있다(§6 참조).

#### 10.3.6 프로젝트 액션 — 화면 하단

- **수정하기** — 인라인 편집 모드 진입. 단계 추가·삭제·제목 수정·드래그로 순서 변경.
- **삭제** — 휴지통 버튼. 확인 다이얼로그 후 영구 삭제(되돌릴 수 없음).
- **목록으로** — 상세 모드 종료, 목록 모드로 복귀(`navHistory` 초기화).

### 10.4 다른 탭과의 연결

전체 탭은 다른 모든 탭과 데이터를 주고받는 허브다.

| 출발 | 도착 | 트리거 |
|---|---|---|
| 홈 (확정 블록) | 전체 탭 (목록) | "확정하기" 클릭 → 신규 카드 추가 + /all 이동 |
| 전체 (목록) | 홈 (시간 설정) | 카드의 "시작 →" 또는 상세의 "시작" |
| 전체 (상세) | 홈 (시간 설정) | 단계의 "시작" |
| 전체 (상세) | 캘린더 | 단계 일정 배정(§11.5) |
| 전체 데이터 | 리포트 | 진행률 · 시간 누적 집계(§12.4) |
| 헤더 "오늘 N분" | — | 전체 탭의 모든 단계 `time_spent` 중 오늘 분량 합산 |

### 10.5 데이터 흐름 — 화면 ↔ DB

| 화면 동작 | DB 변경 | 즉시 갱신되는 곳 |
|---|---|---|
| 카드 진행률 표시 | (조회만) | 진행률 바, 다음 단계 박스 |
| 단계 완료 체크 | `steps.done = true` | 진행률 카드, 카드 % 바, 메트릭 카운트 |
| 하위 단계 완료 체크 | `steps.done = true` (하위 행) | 본 단계 자동 완료 여부 재계산 |
| "시작" → 타이머 종료 | `steps.time_spent += seconds` | 헤더 "오늘 N분" 필 |
| "2단계 쪼개기" | 신규 `decompositions` row + 하위 `steps` 다수 insert | 펼친 카드의 하위 박스 |
| 프로젝트 삭제 | `projects` row + cascade로 `steps`, `decompositions`, `schedule_assignments` 삭제 | 목록에서 카드 사라짐 |
| 프로젝트 수정(단계 변경) | 새로운 `decompositions.round` 추가 + 신규 `steps` 세트 | 상세 모드 단계 리스트 |

### 10.6 설계 원칙 적용

§1.2의 3대 가치가 이 탭에서 어떻게 드러나는가:

| 가치 | 적용 |
|---|---|
| **의미 우선** | 카드는 "이름·진행률·다음 행동" 세 가지만 보인다. 메타데이터(생성일·총 시간·태그 등)는 의도적으로 숨김. |
| **사용자 주도** | 카드 클릭(상세 진입), 펼치기, 2단계 쪼개기, 수정, 삭제 — 모든 행동이 사용자 명시적 선택. AI가 자동으로 단계를 추가하거나 정리하지 않는다. |
| **설명 가능성** | 진행률 % · D-Day · "다음 단계"는 모두 결정론적 계산. 사용자가 "왜 이게 다음이지?"를 한 번에 추적 가능. |

## §11. 캘린더 · 일정 관리

### 11.1 개요

확정된 프로젝트의 단계를 특정 날짜에 배정하고, 주간 또는 월간 뷰로 일정을 조망하는 탭이다.

### 11.2 주간 뷰

- **상단 7일 스트립**: 요일명 + 날짜 원형. 오늘은 주황색 강조, 선택 날짜는 검정 배경.
- **하단 할 일 리스트**: 선택 날짜에 배정된 단계를 카드 형태로 표시.
  - 각 카드: 프로젝트 아이콘 + 단계 제목 + 프로젝트명
  - 순서 변경: ▲/▼ 버튼으로 우선순위 조정
- **오늘로 이동 버튼**: 날짜 네비게이션 후 오늘로 빠르게 복귀.

### 11.3 월간 뷰

- **7×6 그리드**: 날짜 셀에 배정된 할 일을 칩(chip)으로 표시.
- **프로젝트별 색상 코딩**: 각 프로젝트의 고유 색상으로 칩 배경 표시.
- **마감일 표시**: 프로젝트 마감일이 있는 셀에 D-Day 뱃지.
- **오버플로우**: 3개 초과 시 "+N개" 텍스트.
- **이전 달 / 다음 달** 네비게이션 화살표.

### 11.4 뷰 전환

- 주간/월간 뷰 토글 버튼 (세그먼트 컨트롤 형태).
- 뷰 상태는 세션 내 유지.

### 11.5 일정 배정

- 프로젝트 상세 화면에서 개별 단계를 날짜에 드래그하거나 날짜 선택으로 배정.
- 배정 데이터는 `schedule_assignments` 테이블에 저장.

## §12. 리포트 · 진행률 추적

### 12.1 개요

리포트 탭은 사용자의 집중 활동을 시각적으로 요약하여 동기를 유지하고, 개선점을 제시한다.

### 12.2 주간 요약 카드

- **4주 추이 차트**: 최근 4주간 집중 시간 막대 그래프 + 완성 수 표시.
- **이번 주 하이라이트**: 총 집중 시간 · 완료 단계 수 · 평균 세션 시간.

### 12.3 자동 코멘트

AI 또는 규칙 기반으로 주간 활동에 대한 피드백을 자동 생성한다:

- **잘한 점**: 예) "이번 주 3일 연속 집중했어요! 꾸준함이 인상적이에요."
- **아쉬운 점**: 예) "수요일 이후 집중 시간이 줄었어요. 작은 단계부터 시작해볼까요?"

### 12.4 프로젝트별 분석

- **프로젝트별 시간 분배 바**: 각 프로젝트에 투자한 시간을 색상 바로 표시.
- **프로젝트별 완료율**: 전체 단계 대비 완료된 단계 백분율.

### 12.5 나의 기록 (Account 탭)

- **누적 통계**: 총 완료 수 · 총 집중 시간 · 연속 일수.
- **히스토리**: 완료된 프로젝트 목록 (아코디언 형태).

## §13. 기술 스택과 캐시 계층

### 13.1 스택 요약

- **프론트** — React 18+ (SPA), TailwindCSS, Vite.
- **백엔드** — Express.js · Node 20 · Anthropic SDK · zod.
- **데이터** — Supabase (PostgreSQL + RLS).
- **AI** — Claude Sonnet 4 (API), prompt caching (ephemeral, 5분 TTL).

### 13.2 캐시 3계층

| 계층 | TTL | 저장 대상 | 효과 |
|---|---|---|---|
| ① **브라우저 세션** | 탭 세션 | 최근 결과 트리 · 펼친 단계 · 드래프트 · 마지막 입력값 | 왕복 시 즉시 복원, 네트워크 0회 |
| ② **Supabase 영구** | 영구(삭제 전) | projects · steps · decompositions · reasoning_logs · schedule_assignments | 단일 진실 원본, 재접속 복원 |
| ③ **Anthropic Prompt Cache** | 5분 (ephemeral) | 시스템 프롬프트 · 6 신호·3 규칙 · JSON 스키마 · 분석·분해 통합 지시문 (모든 호출 공유 접두부) | 반복 호출 시 입력 토큰 90% 할인 · TTFB 단축 |

### 13.3 비용 시나리오 (1회 분해 기준, 대략)

- **DECOMPOSE 최초 호출** (입력 ~3K, 출력 ~1.5K 토큰): 캐시 미적중 ≈ ₩7
- **DECOMPOSE 반복 호출**: 접두부 캐시 적중 시 ≈ ₩1 내외
- **REFINE 반복 호출**: 대부분 캐시 적중 → 호출당 ≈ ₩1 내외

이전 설계 대비 이득: UNDERSTAND 별도 호출(~300토큰)이 사라지고 DECOMPOSE 하나로 통합되면서 네트워크 왕복 1회가 절약되고, 모든 호출이 동일 접두부를 공유해 캐시 적중률이 높아진다. 정확 수치는 월말 사용량 모니터링으로 갱신.

## §14. 데이터 모델

### 14.1 핵심 테이블

- **users** (id, email, created_at)
- **projects** (id, user_id, raw_input, primary_type, secondary_tags, goal, current_phase, color, start_date, due, is_single, scale, template_id, template_name, created_at)
- **decompositions** (id, project_id, round, parent_decomp_id, trigger, created_at)
- **steps** (id, decomposition_id, parent_step_id, order_idx, title, description, guide, estimated_minutes, done, time_spent, boundary_signal)
- **reasoning_logs** (id, decomposition_id, signals_used, analysis_json, raw_llm_json)
- **schedule_assignments** (id, step_id, user_id, date, slot_start, slot_end)

### 14.2 트리 구조와 버전

- `steps.parent_step_id`로 2차 쪼개기 결과를 트리로 표현.
- `decompositions.round`로 1차/2차/재분해 차수 추적, `parent_decomp_id`로 계보 유지.
- 확정은 덮어쓰지 않고 새로운 `round`로 버전 저장 → 실수해도 이전 분해 복원 가능 (최대 3개 이전 버전).

### 14.3 프로젝트 색상

각 프로젝트에 고유 색상이 배정된다. 캘린더 칩, 리포트 바, 전체 목록 아이콘 등에서 프로젝트를 시각적으로 구분하는 데 사용한다.
