# 인증 시스템 개선 제안

## 현재 구조 요약

로그인/회원가입은 프론트엔드에서 Supabase Auth로 직접 처리하고, 백엔드는 발급된 access_token을 검증만 하는 구조. 인증 관련 코드는 다음 4곳에 분산.

| 위치 | 역할 |
|---|---|
| `frontend/src/pages/LoginPage.tsx` | 로그인/회원가입 폼 (`signInWithPassword`, `signUp`) |
| `frontend/src/components/SessionGuard.tsx` | 라우트 가드. `onAuthStateChange` 구독 |
| `frontend/src/services/projects.ts` | API 호출 시 `Authorization: Bearer` 헤더 첨부 |
| `backend/src/middleware/auth.ts` | 토큰 검증 + `public.users` upsert |

전체적으로 동작은 하지만, 누락된 기능과 손볼 부분이 있어 우선순위별로 정리.

---

## P0 — 기능 누락 (즉시)

### 1. 로그아웃 기능 부재

`signOut` 호출이 프론트엔드 어디에도 없음. 한 번 로그인하면 사용자가 빠져나올 방법이 없음.

**작업:** `AccountPage`(또는 적절한 위치)에 로그아웃 버튼 추가.
```ts
await supabase.auth.signOut();
// SessionGuard의 onAuthStateChange가 잡아 자동으로 /login 이동
```

### 2. 이메일 인증 미적용

현재 Supabase의 "Confirm email" 옵션이 OFF라 가입 즉시 세션이 생성됨. 이메일 소유권 확인이 되지 않아 오타 계정·도용 가입·재설정 메일 무력화 등의 문제가 있음.

**작업:**
- Supabase 대시보드 → Authentication → Providers → Email → **Confirm email 토글 ON**
- URL Configuration의 Site URL과 Redirect URLs에 `http://localhost:5173`(개발), 배포 도메인 등록
- `signUp` 호출에 `options.emailRedirectTo` 추가
- LoginPage의 `data.session`이 null인 분기 메시지를 "확인 메일 발송" 안내로 정상화
- "Email not confirmed" 에러 한국어 매핑 + `supabase.auth.resend()` 재발송 버튼

**운영 주의:** Supabase 기본 SMTP는 시간당 약 4통 제한. 배포 전 커스텀 SMTP(Resend, SendGrid, Mailgun 등) 연결 필요.

> 참고: Confirm email은 가입 시 1회만 적용됨. 매 로그인마다 인증을 강제하지는 않음.

## P1 — 버그/규약 위반

### 3. 폼이 `react-hook-form + zod` 미사용 (규약 위반)

CLAUDE.md 명시 규약 위반. LoginPage가 `useState` 4개로 폼을 직접 관리 중.

**작업:** 입력 검증을 zod 스키마로 정의하고 RHF로 마이그레이션.

### 4. `LocationState.from` 타입 오류 (LoginPage.tsx:5-7)

```ts
type LocationState = { from?: Location };  // ← DOM 전역 Location으로 잡힘
```

**작업:** `react-router-dom`의 `Location`을 명시적으로 import.

### 5. 세션 확인 중 폼 깜빡임 (LoginPage.tsx:20-24)

`getSession` 비동기 응답 전까지 `isSignedIn=false`라 이미 로그인된 사용자도 한 프레임 동안 폼이 노출됨.

**작업:** `authChecked` 상태 추가, 로딩 중엔 빈 화면/스피너 렌더링.

### 6. 중복 이메일 가입 감지 불가 (LoginPage.tsx)

Supabase는 보안상 중복 이메일에 에러를 던지지 않고 `data.user.identities`를 빈 배열로 반환. 현재 코드는 이걸 "이메일 인증 대기"와 동일하게 처리.

**작업:**
```ts
if (data.user && data.user.identities?.length === 0) {
  // "이미 가입된 이메일이에요. 로그인해주세요." 분기
}
```

## P2 — UX

### 7. 에러 메시지 영문 그대로 노출 (LoginPage.tsx:38)

`Invalid login credentials`, `User already registered` 등이 사용자에게 노출됨.

**작업:** 한국어 매핑 함수 추가.

### 8. 회원가입 성공 분기 메시지 부정확 (LoginPage.tsx:49)

`data.session`이 null인 경우 "회원가입을 바로 완료하지 못했어요"는 정상 흐름인데 실패처럼 읽힘.

**작업:** "확인 메일을 보냈어요. 메일함의 링크를 눌러 가입을 완료해주세요." 로 수정.

### 9. `autocomplete` 속성 누락

비밀번호 매니저가 동작하지 않음.

**작업:** 이메일 input에 `autoComplete="email"`, 비밀번호 input에 모드별로 `"current-password"` / `"new-password"`.

### 10. 이메일 정규화 부재

대소문자/공백 차이로 같은 사용자가 다르게 처리될 위험.

**작업:** 제출 직전 `email.trim().toLowerCase()`.

### 11. 부재 기능

- 비밀번호 보기 토글
- 비밀번호 찾기/재설정 링크 (`supabase.auth.resetPasswordForEmail`)
- 회원가입 시 비밀번호 재확인 필드

## P4 — 구조 리팩터링

### 13. `AuthProvider` + `useAuth` 훅으로 중복 제거

LoginPage와 SessionGuard가 각자 `getSession`을 호출. AccountPage에 로그아웃을 추가하면 패턴이 또 복제됨. 컨텍스트로 모으면 5번(폼 깜빡임)도 자연스럽게 해결됨.

### 14. `redirectPath` 검증 (LoginPage.tsx:12)

`location.state.from.pathname`을 무검증 사용 중.

**작업:** "/"로 시작하고 "//"로 시작하지 않는지 화이트리스트 체크.

### 15. `status` enum 단순화

`"idle" | "submitting" | "success" | "error"` 4개 중 실제 분기에 쓰이는 건 `submitting`(disable)·`error`(색상)뿐. `submitting: boolean` + `error: string | null`로 충분.

### 16. 에러 로깅 부재

`error` 발생 시 `console.error` 또는 모니터링 송신 없음. 운영 디버깅 어려움.

## P5 — 백엔드

### 17. 보호 API 요청마다 users upsert (backend/auth.ts:34-37)

매 요청마다 `public.users` upsert가 실행됨. 트래픽이 늘면 불필요한 쓰기 부하.

**대안:**
- (권장) Supabase DB trigger로 `auth.users` insert 시 `public.users` 자동 생성 → 미들웨어에서 upsert 제거
- (간이) 메모리 `Set<userId>` 캐시로 이미 본 유저는 skip

### 18. `email ?? ""` 빈 문자열 fallback (backend/auth.ts:36)

컬럼이 NOT NULL이라면 방어 차원이지만, 빈 문자열 저장이 의도된 건지 확인 필요. nullable로 바꾸는 편이 깔끔할 수 있음.

## 선택 검토

### 19. 봇 방어 (hCaptcha)

Supabase가 native 지원. 가입 폼에 추가하면 메일 발송 한도 보호에도 도움.

### 20. 비밀번호 정책 강화

현재 클라이언트 `minLength=6`. 정책 검토 후 8자 이상 + 숫자/문자 혼합 등 검토.
